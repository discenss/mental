# Mental Club — Runbook (локальный запуск + хендофф)

Живой запуск бэкенда и Telegram-бота. Рассчитано на то, чтобы **другая сессия/модель**
могла поднять проект с нуля без контекста предыдущего чата.

> Архитектура и модель данных: [mental-architecture.md](mental-architecture.md).
> Каноничная спека продукта — стандарт «Архитектура модулей v1.0» (docx, вне репо).

### Токены Telegram — ВАЖНО
- `@discens_test_bot`, token начинается на `1061357234…` — **тестовый**, его и используем для mental.
  Лежит в `rhythmos/bot/.env` как `BOT_TOKEN`.
- Прод rhythmos — **другой** токен (`7925068889…`, в `rhythmos/backend/.env` как

  будет `TelegramConflictError: Conflict` (см. §5).

## 4. Как пользоваться ботом

**Вход для новых/сброшенных (2026-07-15):** `/start` смотрит `api.user_enrollments`.
Если записей нет (новый или после `/reset`) — НЕ показываем полное меню, а мягкий экран
`WELCOME_NEW` + инлайн-кнопка «🚀 Начать» (`onboard_start`). По «Начать» → интро сервиса
(`ONBOARD_INTRO`: «сервис психоподдержки, не диагноз») + выбор (инлайн):
«🧭 Подобрать программу (пройти опрос)» (`go_intake`) / «📚 Выбрать программу самому» (`show_modules`).
**Полное меню на онбординге НЕ показываем** (Сегодня пусто, Спросить ИИ не о чем, Подобрать/Модули
дублируют инлайн-выбор) — `cb_onboard_start` шлёт `ReplyKeyboardRemove`. Полное меню (`main_menu_kb`)
появляется только ПОСЛЕ записи на программу (в `cb_enroll`).
`/reset` завершается тем же стартовым экраном (снимает меню через `ReplyKeyboardRemove`).
Вернувшийся (есть записи) — сразу `WELCOME` + полное меню.

`/start` (вернувшийся) → главное меню (reply-клавиатура):
- **📅 Сегодня** — дневной flow в ДВЕ сессии (см. §8d): 🌅 утро (открыть: утр. маркеры +
  фокус + задание) и 🌇 вечер (закрыть: статус задания + квиз + веч. маркеры + рефлексия →
  следующий день). Гейт «1 день/сутки» живёт в боте.
  Финал дня (`_finish_day`, по `complete_day.status`): день 1–6 → одно сообщение `DAY_DONE`
  («✅ День завершён… возвращайтесь завтра», без дубля show_today); день 7 → предложение
  самопроверки; тест-режим → сразу следующий день.
  eid берётся из FSM, а если потерян (рестарт бота) — по активной записи пользователя.
  Рефлексии (свободный текст): можно **текстом или голосом** (Whisper) + кнопка «⏭ Пропустить»
  (подпись «Ответьте текстом или надиктуйте голосовым 🎤»; `/skip` тоже работает как fallback).
  Распознанный голос бот показывает эхом «🎤 …» (`voice.message_text`) — подтверждение приёма,
  иначе казалось, что «не подхватилось» (сам Whisper работал, просто не было обратной связи).
- **🧭 Мой путь** — сводка/статистика прогресса: модуль, неделя N/6 · день M/7,
  пройдено дней X/42, зоны пройденных недель. (Это и есть «моя статистика».)
- **⚙️ Настройки** — ежедневное напоминание: вкл/выкл, время (ЧЧ:ММ), часовой пояс (IANA).
- **📔 Дневник** — «Мой дневник» (§8): индекс по датам (кнопка на каждый день с числом
  записей + «📝 Добавить заметку»); тап по дню → записи этого дня, «⬅️ К дневнику» назад.
- **🔄 Начать заново** — то же, что `/reset` (сброс с подтверждением).
- **🤖 Спросить ИИ** — свободный вопрос (текст/голос, простая модель).
- **🧭 Подобрать маршрут** — интейк (30 вопросов, шкала 0–4) → ведущее направление + результат.
  UI: вопросы редактируются **на месте** (не копятся — `edit_text` по одному сообщению,
  `iq_msg_id`); шкала — иконки-градация `SCALE_ICONS {0:○ 1:◔ 2:◑ 3:◕ 4:●}` + подпись,
  без ведущей цифры.
  - Если по направлению есть готовый модуль → кнопка «▶️ Начать: <имя модуля>».
  - Если модуля ещё нет (8 из 10 направлений) → кнопки доступных модулей (BOUND/REAL),
    чтобы **не было тупика**. См. §6.
- **📚 Модули** — каталог, ручной выбор модуля.
- **🧪 Тест-режим** (только TESTER_IDS):
  - **✍️ Пройти вручную (без ожидания)** — настоящий дневной flow с реальными ответами и
    рефлексиями, но без суточного гейта → наполняет `daily_entries`/дневник → ИИ-«Итог модуля»
    становится осмысленным. Это способ проверить характеристики по-настоящему.
  - **⏩ перемотка** день/неделя/вся программа × пресеты (лучшие/худшие/случайно) —
    авто-ответы, дамп выдачи движка. Самопроверки авто-считаются (зоны/critical реальные),
    рефлексии пустые (дневник/ИИ-итог не засоряются).
- **/reset** — полный сброс прогресса (программы, ответы, дневник, самопроверки) и старт заново.
  Идентичность (User) сохраняется. Требует подтверждения. Эндпоинт `POST /api/v1/users/reset`.

День 7 → недельная самопроверка (10 вопросов) → зона GREEN/YELLOW/RED + critical-логика.
Завершение модуля → кнопка «🪞 Итог модуля (ИИ)» (аналитическая модель).

## 5. Типовые проблемы

| Симптом | Причина / фикс |
|---|---|
| `TelegramConflictError: Conflict … terminated by other getUpdates` | Запущено ≥2 копии бота. Найти: `ps aux \| grep "bin/python main.py"`. Убить лишние **по PID** (`kill -9 <pid>`). |
| `pkill -f "…main.py"` рубит саму сессию (exit 144) | Паттерн matchит и лаунчер-shell. **Не использовать `pkill -f` по main.py** — только `kill <pid>`. |
| Бот не видит BOT_TOKEN | `bot/config.py` грузит `bot/.env` через `load_dotenv`. Проверить, что файл на месте и `BOT_TOKEN=` заполнен. |
| ИИ отвечает заглушкой | Нет `OPENAI_API_KEY` → `settings.llm_enabled=False`. Не падает, отвечает graceful. |
| ADD COLUMN NOT NULL падает на непустой БД | В миграции проставить `server_default=…` (так уже сделано для refs/min_hits/mode). |
| «Залип на шаге дня» (кнопки/ввод не срабатывают) | Бот на MemoryStorage → при перезапуске теряется FSM-состояние дня. Восстановление: любое сообщение ловит `start.fallback` → `show_today` пересобирает текущую сессию с бэкенда (morning/evening по `*_done`); «📅 Сегодня» делает то же. Глубокий фикс — персистентный FSM-storage (Redis/SQLite) при деплое. |

Логи текущего живого прогона (эта сессия) — в scratchpad: `mental_backend.log`, `mental_bot.log`.

## 6. Интейк → модуль: маршрутизация и «куда идти» (фикс 2026-07-15)

**Проблема:** реализованы только 2 модуля из 10 направлений (BOUND, REAL). Если интейк
выдавал ведущим одно из 8 будущих направлений — `leading_module=None` → кнопки «Начать» не
было → тупик. А когда модуль был (BOUND/REAL), кнопка подписывалась именем *направления*
(`name_leading`), которое ≠ имени *модуля* (у REAL особенно: направление «Реализация, смысл
и новый ориентир» vs модуль «Найти себя и вернуть смысл»).

**Фикс:**
- `services/intake.py` → ответ `/intake/submit` теперь содержит:
  - `leading_module_name` — реальное имя готового модуля (или `null`);
  - `available_modules` — `[{code, name, is_leading, is_focus}]`, все готовые модули.
- `/enroll` возвращает `name` (человекочитаемое имя модуля).
- Бот (`keyboards.intake_result_kb`, `texts.intake_result`, `handlers/intake.py`):
  ведущий модуль — первой кнопкой по его настоящему имени; если модуля нет — кнопки
  доступных модулей (фокус-направления помечены ⭐). Тупик исключён.
- `handlers/start.py` cb_enroll: подтверждение по имени модуля + явное «куда дальше»
  (формат 6 недель, кнопка «📅 Сегодня»).

**Наименование (открытый вопрос):** направление REAL и модуль REAL называются по-разному.
Пока показываем: интерпретацию — по имени направления, кнопку/подтверждение — по имени модуля.
Согласовать единое имя при желании владельца.

### Экран результата интейка (доработка 2026-07-15, v2)
Логика «одна рекомендация + другие маршруты» (по пожеланию владельца):
- CTA называет **конкретную программу** (не абстрактный «ведущий маршрут»):
  «Рекомендуем начать с программы «<имя модуля>»».
- Кнопки (`keyboards.intake_result_kb`): ровно две —
  1. если модуль готов → активная «▶️ Начать: <имя модуля>» (`enroll:CODE:normal`);
     если не готов → неактивная «🔒 <имя направления> — скоро» (`route_soon` → тост);
  2. «🗂 Другие маршруты» (`show_modules` → пикер доступных программ = как «📚 Модули»).
- Контент: `intake.yaml.must_include` переписан без жаргона «ведущий маршрут»
  → «Начать можно с рекомендованной программы… выбор всегда остаётся за вами».
  (После правки контента — `python -m scripts.load_content`.)

**Кто даёт рекомендацию:** НЕ ИИ. Интейк-роутер полностью детерминирован
(`services/intake.py`): вопрос → направление (`direction_code`) → модуль (`module_code`);
баллы по направлениям → ранжирование по `tie_priority` → ведущее + 2 фокуса; тексты —
готовые интерпретации из `intake.yaml`. OpenAI участвует только в «Спросить ИИ»,
«Итог модуля» и распознавании голоса (Whisper).

## 8. «Мой путь» + «Мой дневник» (реализовано 2026-07-15)

Раньше в боте не было ни экрана прогресса, ни дневника — модель `JournalEntry` была,
но пустая и ни к чему не подключена. Добавлено:

**Backend:**
- `complete_day` → `_sync_journal`: непустые рефлексии дня пишутся в `journal_entries`
  (`source_type='reflection'`), идемпотентно (перезапись записей этого дня при повторе).
  Режим `mode='test'` дневник НЕ засоряет. По спеке §8 в дневник идут рефлексии/задания/
  выводы/финальный продукт/заметки, НЕ идут маркеры/техстатус/квизы/самопроверка.
- `POST /api/v1/users/enrollments` → записи пользователя (для «Мой путь»/«Сегодня» после рестарта).
- `GET  /api/v1/enrollments/{eid}/status` → модуль, неделя/день, статус, зоны недель.
- `POST /api/v1/journal/list` {provider,provider_user_id[,module_code]} → записи дневника.
- `POST /api/v1/journal` {…, text} → личная заметка (`source_type='note'`).

**Bot:** `handlers/progress.py` (Мой путь; Дневник — индекс по датам `_group_by_day` →
`jday:<YYYY-MM-DD>` → записи дня, `jindex` назад; добавление заметки `note_add`/`NoteStates`
с возвратом в индекс; `resolve_eid`), кнопки в `main_menu_kb`, форматтеры `texts.path_status`,
`texts.journal_index` / `texts.journal_day` / `texts._date_disp`.

**Ещё не в дневнике (TODO):** финальный продукт модуля и авто-сохраняемая формула
REAL W6D6 (`quiz.saves_free_text`) — пока не эмитятся в `journal_entries`; квиз в боте
всегда с вариантами (свободный текст для W6D6 не реализован).

**Правило дневника:** `progression._sync_journal` пишет только НЕПУСТЫЕ рефлексии
(идемпотентно по дню). Гарды по `mode` нет — вместо этого авто-прогон тест-режима
шлёт `reflection=[]` (ничего не пишется), а ручной тест-прогон с реальным текстом —
наполняет дневник. Так тест-режим можно использовать для проверки дневника и ИИ-итога.

## 8a. Сброс данных + ручной тест-прогон (2026-07-15)

**Сброс (`/reset`, `POST /api/v1/users/reset`):** `identity.reset_user(db, user_id)` удаляет
все `enrollments` пользователя и их дочерние строки (daily_entries, selfcheck_results,
flag_accumulator, final_product_instances, postmodule_results) + `journal_entries` и
`intake_results` (по user_id). User + UserIdentity остаются. В боте — команда `/reset`
с подтверждением (`reset_yes`/`reset_no`), затем `state.clear()` и главное меню.

**Ручной тест-прогон:** кнопка «✍️ Пройти вручную» (`tmanual`) в тест-меню запускает обычный
`show_today` для test-enrollment. Суточный гейт снят: `flow.show_today` пропускает
`COME_BACK_TOMORROW`, когда `mode=='test'` (mode берётся из FSM, ставится при enroll и в
`resolve_eid`). После самопроверки авто-переход к следующей неделе не делается — тестировщик
жмёт «📅 Сегодня» (мелочь, можно доработать).

**ИИ-разбор недели (2026-07-15):** после недельной самопроверки — кнопка «🪞 Разбор недели (ИИ)»
(`week_insight`) → `POST …/week-insight` → `ai.week_insight` (analytics-тир, промпт `WEEK_SYSTEM`):
берёт последнюю пройденную неделю (max `SelfcheckResult.week_n`) — её зону + рефлексии этой недели
(`daily_entries.reflection_answers` за неделю) → тёплый отклик 2–3 абзаца, без диагнозов.
Недельная зона/рекомендация по-прежнему детерминированные; ИИ — дополнительный разбор поверх.

**Кто выдаёт «Итог модуля»:** это ИИ (analytics-тир, `ai.module_insight`) — читает
`selfcheck_results.zone` (реальные, даже в авто-прогоне) + `daily_entries.reflection_answers`
+ финальный продукт. Недельная же характеристика (зона/рекомендация/critical) —
ДЕТЕРМИНИРОВАННАЯ (`scoring.score_week`), не ИИ.

## 8b. Настройки + напоминания + статистика (2026-07-15)

Перенесено из rhythmos «по схеме», но лаконично (rhythmos-планировщик на 1100 строк —
адаптивный, тут не нужен).

**Данные:** в `users` — `reminder_enabled` (bool) + ТРИ слота времени: утро/день/вечер.
Утро/день — `reminder_{morning,afternoon}_{hour,minute}` (миграция `d2b3c4e5f6a7`, дефолты
10:00/14:00); вечер исторически = `reminder_hour/minute` (20:00). Дедуп по слотам:
`last_morning_date` / `last_afternoon_date` / `last_reminded_date`. Таймзона — `users.timezone`
(дефолт `Europe/Riga`). Требует `apscheduler` + `pytz` (есть в venv rhythmos). Слоты описаны
в `settings.SLOTS = {slot: (hour_attr, minute_attr, date_attr)}`.

**Бэкенд** (`services/settings.py`):
- `get_settings` / `update_settings` (валидирует час/минуту/таймзону).
- `due_reminders(db)` — КАНДИДАТЫ на напоминание сейчас (по слотам): активная normal-запись,
  день не закрыт, есть telegram-идентичность; по каждому слоту — локальное время ≥ времени слота
  и слот сегодня ещё не доставлен. **НЕ стемпит** (важно!) — иначе при недоступном боте слот
  «съедался» бы без доставки (был баг: утро помечалось в ~10:00, когда бот заморожен, и днём
  приходил только afternoon). Кандидат повторяется каждый тик, пока не доставлен.
- `mark_reminded(db, provider_user_id, slot)` — стемпит `last_<slot>_date` = локальная дата.
  Вызывается ботом ТОЛЬКО после фактической доставки (или при Forbidden/BadRequest — чтобы не
  долбить недоступный чат). Так напоминание не теряется при сбое/паузе бота.
- **Напоминания нельзя отключить** (по решению владельца) — можно менять только время/пояс.
  Переключателя вкл/выкл нет; `due_reminders` не фильтрует по `reminder_enabled` (колонка мертва).
- Эндпоинты: `POST /users/settings`, `POST /users/settings/update`, `POST /reminders/due`
  (внутренний, для планировщика бота — в проде стоит закрыть).
- `GET /enrollments/{eid}/status` дополнен `days_completed` / `days_total` (=42).

**Бот:**
- `scheduler.py` — APScheduler (AsyncIO, UTC), джоба раз в минуту: `api.reminders_due()` →
  `bot.send_message` с `texts.reminder_nudge`. Стартует в `main.py` (`start_scheduler(bot)`),
  гасится в finally. Логика — на бэкенде, здесь только отправка.
- `handlers/settings.py` — «⚙️ Настройки»: время каждого слота (`set_time:<slot>`,
  FSM `SettingsStates.time` + `set_slot`, ЧЧ:ММ) / таймзона (FSM `SettingsStates.tz`, IANA).
  Переключателя вкл/выкл НЕТ (напоминания всегда включены). `texts.reminder_nudge` — свой текст
  на каждый слот. `scheduler._tick`: send → `api.mark_reminded` (стемп после доставки); при
  Forbidden/BadRequest тоже стемпит; при сетевой ошибке НЕ стемпит (повтор в след. тик).
  Нечисловые provider_user_id пропускаются.
- «🧭 Мой путь» показывает «Пройдено дней X/42» (это и есть статистика; отдельной кнопки нет).

**Проверено HTTP:** дефолт 20:00 Europe/Riga; апдейт 09:30; кривая таймзона → 422; `due` для
пользователя с активной записью и временем в прошлом → 1 запись, повторно → 0 (дедуп).

## 8c. Дневной flow v2 + фикс кнопок меню (2026-07-15)

**Баг «кнопка Настройки не работает»:** пока пользователь в состоянии `DayStates.running`
(или Ask/Note/Settings), catch-all message-хендлер этого состояния глотал нажатия reply-кнопок
меню. Фикс: в начале всех stateful message-хендлеров — если `msg.text in keyboards.MENU_TEXTS`,
`state.set_state(None)` + `raise SkipHandler()` → сообщение проходит дальше к меню-хендлеру.
(Работает т.к. flow/ask/… зарегистрированы раньше settings/start; краевые случаи «кнопка апстрим-
роутера из даунстрим-состояния» редки.)

**Возврат в прерванный день:** escape сбрасывает состояние, но ДАННЫЕ дня (`day_steps`,
`day_i`, `day_session`) остаются в FSM + флаг `day_active`. `flow.resume_if_active(target,state)`
ставит состояние обратно в `DayStates.running` и перерисовывает текущий шаг. Вызывается из
«📅 Сегодня» (`menu_today`) и по кнопке «↩️ Продолжить день» (`resume_day`, показывается после
ответа ИИ, если `day_active`). `day_active` ставится в `show_today`, снимается в `_finish_day`.

**Дневной flow (`handlers/flow.py`) переписан:**
- маркеры перекрывают друг друга на месте (`edit_text` по одной карточке), в конце блока
  карточка замораживается в **сводку иконками** `☀️ Утро: ● ◕ ◔ ○ ●` (`_phase_summary`);
- иконки маркеров `keyboards.MARKER_ICONS {Да:● Скорее да:◕ Скорее нет:◔ Нет:○}`, кнопки по 2 в ряд
  (`marker_kb`), без цифр;
- **фокус + задание — одним сообщением** (`focustask`, статус задания там же кнопками);
- квиз — своя карточка, при ответе замораживается в «❓ вопрос ✔️ выбор»;
- недельная самопроверка тоже перекрывается на месте;
- рефлексии — отдельные сообщения (текст/голос/пропустить).

**Первый вход в программу:** после enroll (normal) бот НЕ стартует день сам (иначе настройки
и flow дня делят одно FSM-состояние и конфликтуют — вход в set_time «отключал» кнопки вопроса).
Показываем: подтверждение → `settings_kb` (настроить напоминания по желанию) → отдельная кнопка
«▶️ Открыть первый день» (`start_day`). Хендлер `cb_start_day` сбрасывает состояние и вызывает
`show_today`. Так потоки последовательны, без клеша.

## 8d. Утренняя и вечерняя сессии дня (2026-07-15)

День разделён на две сессии (чтобы не «всё сразу»):
- **🌅 Утро (открыть день):** утренние маркеры + показ фокуса и задания. НЕ спрашивает статус.
- **🌇 Вечер (закрыть день):** статус задания (сделано/частично/не сделано) + квиз +
  вечерние маркеры + рефлексия → продвижение к следующему дню.

**Данные:** `daily_entries.morning_done` / `evening_done` (миграция `e3c4d5f6a7b8`; старые
полные дни проставлены обеими). `entry_date` стемпится при ОТКРЫТИИ (утро).

**Бэкенд (`progression.py`):** `open_day` (утро, без продвижения), `close_day` (вечер +
`_advance`), `complete_day` (обе сразу — для тест-режима/авто-прогона). `get_today` отдаёт
`session: "morning"|"evening"` (вечер, если текущий день уже открыт). Гейт: открыть НОВЫЙ
день нельзя, если сегодня уже закрыли день (`_closed_today` → `done_today` только для morning).
Эндпоинты `POST …/open-day`, `POST …/close-day` (плюс прежний `…/complete-day` для теста).

**Напоминания (session-aware):** утренний слот — только если день не открыт; дневной/вечерний —
пока день не закрыт; если день закрыт сегодня — на сегодня всё. (Проверено HTTP.)

**Клиентский вводный экран недели:** `intro_screen`/`goal`/`key_themes` недели давно есть в модели
и контенте, но в дневном flow не показывались. Теперь `get_today` отдаёт `week_intro`, и бот
показывает его один раз — в начале недели (день 1, утренняя сессия) через `texts.week_intro`,
перед шагами дня. (Также доступен каталожным `GET /modules/{code}/weeks/{n}`.)

**Бот (`flow.py`):** `show_today` строит утренние (`_build_morning_steps`) или вечерние
(`_build_evening_steps`) шаги по `session`; `_finish_day` вызывает `open_day` (→ `DAY_OPENED`,
«вернитесь вечером») или `close_day` (→ `DAY_DONE`/самопроверка). В тест-режиме сессии идут
подряд без ожидания. Гейт (`done_today`) блокирует только повторное открытие нового дня.

## 8e. Аудио, постмодуль, финальный продукт, §7/§12 добивка (2026-07-17)

Закрытие критичных расхождений со «Стандартом архитектуры v1.0» (см. `compliance-audit.html`).

### Аудио-практики (§8.4, §13)

Раньше аудио было текстовой строкой в фокус-блоке. Теперь — реальный медиа-пайплайн.
- **Хранение:** файлы на диске в `content/audio/` (имя = код, напр. `AUDIO_BOUND_W1_A2.mp3`),
  в БД — только метаданные. Модель `AudioAsset` расширена полями `media_filename`, `mime`,
  `duration_sec`, `size_bytes`, `tg_file_id` (миграция `096cebe99ba6`). Лоадер читает `file:`
  из `audio_map` в YAML.
- **Отдача:** `get_today` отдаёт аудио-блок **только если файл реально есть** (`media_filename`),
  чтобы не было битых кнопок. Эндпоинты: `GET /api/v1/audio/{code}/file` (сам файл по HTTP,
  `accept-ranges` — пригодится для macOS-клиента) и `POST /api/v1/audio/{code}/tg-file-id`
  (кэш Telegram file_id).
- **Бот (`flow.py`):** в утренней сессии, после фокуса/задания — отдельный шаг с кнопкой
  **«🎧 Слушать аудио»** (`kind:"audio"`). По нажатию: есть `tg_file_id` → шлём его; иначе
  `FSInputFile` из `content/audio/` → ловим `file_id` → кэшируем на бэке. Путь — `config.AUDIO_DIR`.
- **Долги:** тестовый файл `ANXweek1day3-4.mp3` привязан к `AUDIO_BOUND_W1_A2` (модуля ANX нет);
  у остальных 23 записей `file`/`theme` пустые (аудиоскрипты не наполнены). Reload контента
  (`load_content`) стирает `tg_file_id` (delete+recreate) — перекэшируется при 1-й отправке.
  `.gitignore` для `content/audio/*.mp3` пока не добавлен.

### Постмодульный тест REAL в normal-флоу (§15)

Раньше вызывался только в тест-режиме. Теперь доступен реальному пользователю.
- **Backend (`postmodule.py`):** новый `get_questions()` отдаёт 21 вопрос (7 тем × 3) **без весов**
  (§16); для BOUND (flags) — пусто. `run_test`/`run_flags` обогащены `module_name` и `topic_name`.
  Эндпоинт `GET …/postmodule-questions`.
- **Бот:** на экране завершённого модуля — кнопка **«🧭 Смежные направления»**. REAL: 21-вопросный
  опрос (FSM `PostmoduleStates`, префикс `pm:`) → сабмит → рекомендация. BOUND: сразу по флагам.
  Рендер `texts.postmodule_result` — без баллов/тегов, тон §16.2, «не обязательный переход».
- **Тема без готового модуля** (реализованы только BOUND/REAL): показывается по имени с пометкой
  «направление в подготовке, доступно как ориентир» (честно отражает сильный сигнал).

### Финальный личный продукт + «🏆 Личные достижения» (§14)

Раньше `FinalProductInstance` не создавался вообще. Теперь — пошаговый сбор + хранение.
- **Backend (`finalproduct.py`, новый):** `get_template()` (секции), `save()` (создаёт instance +
  **короткую отсылку** в дневник `source_type=final_product`, идемпотентно), `list_for_user()`,

- **Бот:** на завершении модуля кнопка **«📜 Собрать личный продукт»** → пошаговый сбор по секциям

## 8f. Меню, единый режим, аудио на каждый день (2026-07-17)

### Аудио на каждый день (×48)

Тестовый mp3 раскопирован под **все 48 кодов** (24 BOUND + 24 REAL) в `content/audio/`;
в оба YAML проставлены `file`/`mime`/`duration_sec`/`size_bytes` для каждой записи `audio_map`.
Теперь кнопка «🎧 Слушать аудио» есть на **каждом** дне (слот по дню: 1-2→A1, 3-4→A2, 5-6→A3,
7→FINAL). Это временная заглушка (276 МБ одинаковых копий) — заменить реальными файлами по именам.

### Меню: «программа» вместо «маршрут», активный маршрут

- «🧭 Подобрать маршрут» → **«🧭 Подобрать программу»** (inline: «(пройти опрос)»).
- «📚 Модули»/`show_modules` при активном enrollment (`active`/`selfcheck_due`) больше НЕ дают
  молча записаться заново: показывают `texts.on_active_route` + `active_route_kb`
  («▶️ Продолжить программу» / «🔄 Сменить программу»). «Сменить» объясняет, что нужен `/reset`.
  Гарантия: **один активный маршрут одновременно**.

### Единый режим: тест-режим → кнопка пропуска + `/test`

Убрана параллельная сущность `mode='test'` из меню (кнопка «🧪 Тест-режим», отдельный enrollment).
Теперь один маршрут — одно состояние. Вместо этого:
- **Кнопка «⏭ Не ждать…»** (`skip_wait_kb`, callback `dwait_skip`) на экранах ожидания
  (после утра → сразу к вечерней части; после вечера / «приходите завтра» → к следующему дню).
  Доступна **всем**. Снимает гейт по времени (`show_today(..., force=True)` обходит `done_today`),
  но **вопросы проходятся по-настоящему** — пропускается только ожидание, не содержание.
- **Авто-прогон** (best/worst/random, для отладки/`verify_engines`) сохранён отдельно: команда
  **`/test`** (только тестировщики), работает над ТЕКУЩИМ активным маршрутом — не создаёт вторую
  запись. `cb_enroll` больше не имеет ветки `mode=='test'`; `tmanual` удалён.

**Про `entry_date`:** при пропуске несколько дней маршрута могут иметь одинаковую `entry_date`
(та же календарная дата) — это безвредно. `entry_date` читается только в `_closed_today` (гейт
«1 день/сутки», уже снят кнопкой). Дневник использует своё `created_at` (реальный момент), не
`entry_date`, — записи группируются по фактической дате.
